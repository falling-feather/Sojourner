# 《人的一生》网页游戏 — 开发者文档（需求规范）

本文档用于统一产品意图、技术边界与协作方式，供工程实现与视觉设计（含 Pencil MCP 出稿）共同遵循。前半部分为**需求规范**，后半部分为**当前仓库内的技术实现与数据契约**（随代码演进请同步更新）。

| 读者 | 用途 |
|------|------|

## 2. 产品概述

### 2.1 一句话描述

一款以 **「人的一生」** 为主线的轻量叙事网页游戏：玩家从 **出生** 开始，在若干 **关键时间点** 做出选择与交互，经历不同路径，最终 **走完一生** 并呈现结局。

### 2.2 体验定位

- **内容**：叙事与选择为主，规则易懂，不要求玩家学习复杂系统。
- **表现**：在 **文案风格** 与 **UI 交互** 上追求一定深度与质感（可参考 galgame 的层次感），与「极简小游戏」区分。
- **单局时长**：**3～5 分钟**（**含阅读文本时间**）。此为硬性产品指标，需在文案长度、段落数量、选项数量上统筹。
- **结构规模**：约 **8～10 个时间点**（人生阶段或关键节点），具体命名与划分由内容设计确定，但数量须落在此区间内。

### 2.3 平台与设备

- **核心平台**：**电脑端浏览器**（桌面窗口、鼠标 + 键盘为主要操作方式）。
- **移动端**：**暂不纳入优化范围**（不要求适配触控布局、安全区、手势等）。允许在窄窗口下「能打开」即可，但不作为验收标准。

## 3. 功能需求（规范性陈述）

### 3.1 核心循环

1. 展示当前阶段的叙事文本（可含多段、分页或打字机等形式，由实现与设计共同决定）。
2. 在关键处提供 **玩家选择** 或 **自动推进**（纯叙事）。
3. 根据选择与内部规则更新 **全局状态**（见第 4 节），并可能触发 **概率检定**（见 3.3）。
4. 进入下一叙事片段或下一阶段，直至 **终局**。

### 3.2 分支与长期后果

- 中途选择应能带来 **可感知的路径差异**，例如：
  - 升学类结果（如高考）不同 → 进入不同院校或发展轨道；
  - 职业路径差异 → 影响后续可选事件或叙事变体。
- 分支不必为纯树状爆炸；允许通过 **属性与标志位** 在后续阶段 **汇聚** 到相同大段落，仅用台词或小幅事件区分，以控制制作量与测试成本。

### 3.3 全局事件积分 / 决断点（galgame 式要素）

为增强叙事层次，须引入 **跨场景保留的全局量**，用于：

- **决断点**：记录关键选择或里程碑（布尔或枚举标志），供后续剧情条件判断。
- **全局事件积分（或全局属性）**：数值或档位，反映长期倾向（如压力、健康负债、社会支持等），**不仅在当前节点生效，也可在远期阶段结算**。

上述要素的 **具体字段名、取值范围与是否展示给玩家** 由实现与策划在后续细化，但须满足第 4 节的约束。

### 3.4 随机性与「努力 vs 结果」

- 在关键节点（例如高考、重大面试等）允许 **概率检定**：
  - **努力 / 准备** 应提高 **有利结果的期望**，但 **不保证** 绝对成功；
  - 保留 **小概率** 的 **发挥失常** 与 **发挥超常**，以体现不确定性。
- 随机结果须在叙事上可接受：避免「无解释的一刀切失败」；可通过文案或轻提示说明与压力、状态等相关（**不必向玩家展示公式**）。

### 3.5 延迟后果（累积不利决断）

- 部分选择在 **当下** 可能 **不立即** 表现为剧情重大转折，但会 **累积** 全局属性或标签。
- **示例（产品意图，非唯一实现方式）**：
  - 高中阶段长期熬夜学习、工作后长期熬夜加班 → 累积 **健康相关负担**；
  - 在中老年阶段触发 **疾病** 或 **健康类事件** 的概率或叙事权重上升。
- 该类机制用于体现 **「短期收益 vs 长期代价」**，须与 3.4 的随机性区分：**延迟后果是条件累积触发，不是单次骰子**。

### 3.6 结局

- 游戏须有明确的 **终局呈现**（人生摘要、关键抉择回顾、或开放式总结等，由内容确定）。
- 单局结束后应能 **重新开始** 新一局（是否保留存档非本期必须，由实现决定）。

## 4. 全局状态模型（需求级约束）

实现时必须支持 **可序列化、可推理** 的全局状态，至少包含以下 **概念类别**（具体字段可在技术方案中命名）：

| 类别 | 用途 |
|------|------|
| 数值属性 | 表示可持续变化的压力、健康负债、财力、职业资本等，供效果与条件使用 |
| 标志位 / 决断点 | 记录不可回退或关键分支结果，供后续场景解锁或台词变体 |
| 标签（可选） | 描述习惯或长期行为模式（如「经常熬夜」），便于条件组合 |
| 随机种子（建议） | 支持 **同种子复现** 同一局结果，便于测试与调试 |

**数值范围与平衡** 由策划在内容制作阶段调参；本文档只要求 **机制上** 支持加减、clamp、阈值判断与阶段末结算。

## 5. 内容结构需求（叙事数据）

- 故事内容应与代码 **解耦**：推荐以外置数据（如 JSON/YAML）描述 **阶段 → 场景 → 选择/效果**。
- 每条选择应能声明：
  - 展示文案；
  - 对全局状态的效果（含延迟生效的标签/属性）；
  - 可选的检定规则与结果分支；
  - 下一目标（下一场景或下一阶段）。
- 须具备 **校验手段**（如 schema），避免拼写错误导致卡死或无法到达终局。

## 6. UI / 交互需求（桌面端）

### 6.1 必备界面能力

- **开始/重开**：进入游戏与结束后再开一局。
- **叙事呈现区**：展示主文本；若有多段，须有清晰推进方式。
- **选项区**：在需要时展示选项；选项数量应控制在单屏可理解范围内（建议单次显著选项 **不超过 9 个**，以便可选键盘数字映射）。
- **全局反馈**：在设计上预留 **属性或氛围反馈** 的展示位置（如简易 HUD、图标或简短状态行），是否默认全开由设计决定。
- **检定反馈**：当发生随机检定时，须有 **可感知的反馈**（文案、简短动效或面板），避免静默改状态。

### 6.2 无障碍与操作（桌面端基准）

- 支持 **键盘** 完成核心流程（聚焦、选择、确认）；鼠标点击并行支持。
- 交互元素须有可辨识的 **焦点状态**（具体样式由视觉稿定义）。

---

## 7. 与 Pencil MCP / 视觉协作的约定

### 7.1 分工原则

- **Pencil（.pen 设计稿）**：负责 **视觉风格、布局、组件外观、动效示意**；设计文件可能加密，**仅通过 Pencil MCP 工具** 读写，不直接当作运行时代码。
- **工程实现**：以 **Web 技术栈**（如 React + Vite 等，具体另定）实现可玩版本；实现时 **对照设计 token 与组件规格** 落地，而非把 .pen 嵌入运行时。

### 7.2 交付给视觉侧的输入（本仓库后续可补充）

- 页面清单：标题、主游玩、结局等。
- 组件清单：叙事区、选项列表、HUD、对话框/检定反馈、按钮等。
- 文案为 **数据驱动**，视觉稿需标注 **内容区 vs 装饰区**，避免写死具体剧情文本。

### 7.3 同步方式

- 设计变更后，应同步 **颜色、字号、间距、动效时长** 等 token 或说明文档，保证实现与设计一致。

---

## 8. 非目标（本期明确不做或不要求）

- 不要求联网存档、账号系统或多人对战。
- 不要求真实地理、职业仿真；数值为 **叙事服务**，不追求现实精确模拟。
- 不要求手机端体验优化（见 2.3）。

---

## 9. 验收标准（产品向）

1. 单局可在 **3～5 分钟** 内自然结束（以目标用户阅读速度为基准，允许配置打字速度或跳过）。
2. 流程包含 **8～10 个时间点/阶段** 量级的结构。
3. 存在 **全局属性/决断点** 对中后期剧情或结局的 **可验证影响**。
4. 关键节点存在 **带随机方差** 的结果，且与努力/状态叙事一致。
5. 存在 **延迟后果** 的至少一条完整链路（从早期行为到后期体现）。
6. **桌面端** 主流程可仅通过键盘完成；在主流现代浏览器可运行。

## 11. 数据契约（Zod / JSON）

### 11.1 根对象 `Story`

- `meta`：`title`、`version`、`estimatedMinutes: [min, max]`、`start: { stageId, sceneId }`、`rngSeedDefault?`
- `initial`：全局初始状态（见下）
- `stages[]`：阶段数组，顺序即人生顺序参考

### 11.2 全局状态 `initial`

- `stats`：固定六维——`stress`、`healthDebt`、`support`、`wealth`、`career`、`luck`（除 `healthDebt` 外展示为 0–100 量纲；`healthDebt` 为非负累积值）
- `flags`：`Record<string, boolean>`，决断点
- `tags`：`string[]`，习惯或长期标签（如「熬夜」）

### 11.3 效果 `Effect`（`op` 判别联合）

| op | 含义 |
|----|------|
| `addStat` | 某维 `stat` 加 `value` |
| `setStat` | 设为 `value` |
| `clampStat` | 限制在 `[min, max]` |
| `setFlag` | 布尔标志 |
| `addTag` / `removeTag` | 标签 |

### 11.4 导航 `Next`

- `{ "kind": "scene", "sceneId": "...", "stageId?": "..." }`：同阶段跳转时可省略 `stageId`
- `{ "kind": "stage", "stageId": "...", "sceneId": "..." }`：跨阶段

### 11.5 检定 `check`（`threshold`）

- `modifiers`：可按 `career` 等加权、`stressPenalty`（压力惩罚）、`luckWeight`
- `bands[]`：每条含 `min`–`max`（与结算分匹配）、`label`、`next`、`effects?`
- 结算分 = `clamp( rawRoll + 修正, 0, 100 )`，`rawRoll` 为 0–100 整数骰；**区间并集应覆盖 0–100**（否则运行时报错）

### 11.6 选项 `Choice`

- 无 `check` 时必须提供 `next`
- 有 `check` 时由检定结果的 `band.next` 导航，不要求根级 `next`
- `visibleWhen`（可选）：`tag`、`flag`、`statMax`、`statMin`——用于条件显示（中老年健康分流等）

### 11.7 场景 `Scene`

- `narrative`：字符串段落数组
- `onEnter`：进入场景时应用的效果
- `autoNext`：可选自动跳转（`delayMs` 默认由 UI 使用）
- `isEnding`：为 true 时进入结局界面（仍朗读同场景 `narrative`）

## 12. 运行时行为摘要

1. **开局**：`clone initial` → 对起点场景执行 `onEnter` → 可复现 RNG（`mulberry32` + 字符串 seed 哈希）。
2. **选肢**：先应用选项 `effects`；若有 `check`，弹出检定层，确认后再对目标场景 `onEnter`。
3. **引擎**不持久化存档；**再来一局** 清空为 `initial` 并回到标题。

## 13. UI 组件与键盘（供 Pencil 对齐）

| 区域 | 组件 | 说明 |
|------|------|------|
| 标题 | `TitleScreen` | 「开始」、Enter 开局 |

## 14. 与 Pencil 的对接清单

- **设计稿**：仓库内若有 `.pen`，仅通过 **Pencil MCP** 修改；运行时不依赖 `.pen`。
- **应对齐的 token**：`App.css` 中 `--bg`、`--panel`、`--accent`、`--border` 等 CSS 变量可视为临时 token，Pencil 定稿后可替换为同名或映射表。
- **内容区**：`NarrativeView` 正文、`ChoiceList` 按钮文案来自 `story.json`，视觉稿避免把具体剧情写死在装饰层。

## 15. 构建、校验与测试

```bash
npm install
npm run dev              # 开发
npm run build            # 类型检查 + 生产构建 → dist/
npm run validate:story   # 仅校验 content/story.json
npm test                 # Vitest
npm run lint             # ESLint
```

静态部署：将 `dist/` 发布到任意静态托管（GitHub Pages、Cloudflare Pages 等），**需配置 SPA 回退到 `index.html`**（若使用 history 路由；当前为单页无路由，一般直接可用）。

## 16. 测试策略

- **单元测试**：`effects`、`threshold` 检定确定性（`src/engine/*.test.ts`）。
- **集成级**：`story.test.ts` 对 `content/story.json` 做 `parseStory`。
- **手工**：跑通一局；故意选「熬夜」路径确认中老年分支变化；多次开局观察高考检定差异。
